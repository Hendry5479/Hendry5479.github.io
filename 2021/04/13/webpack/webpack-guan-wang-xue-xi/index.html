webpack
webpack
Publish Date:   2021-04-13

概念

  • webpack是js的静态模块打包工具
  • 处理js程序时会构建dependency graph，对应映射到项目的每个模块，生成多个bundle
  • webpack4不再引入配置文件来打包项目

入口entry

  • entry指示构建denpendency graph的开始
  • 进入entry后，webpack会找出与entry依赖的模块和库
  • 默认值是 ./src/index.js
  • 在webpack.config.js自定义entry
    module.exports = {
entry: 'xxx'
};

输出output

  • 定义输出bundle的位置和名称
  • 默认值是 ./dist/main.js，其他文件在./dist
  • 在webpack.config.js自定义output
    const path = require('path');
module.exports = {
entry: 'xxx',
output: {
  path: path.resolve(__dirname, 'dist'),
  filename: 'xxx'
}
};

loader

webpack只能理解js和json，loader让webpack处理其他类型文件并转为模块，添加到dependency graph

load的两个属性

  • test：识别文件
  • use：定义转换所用的loader

webpack.config.js

const path = require('path');

module.exports = {
  output: {
    filename: 'my-first-webpack.bundle.js'
  },
  module: {
    rules: [
      { 
        test: /\.txt$/, 
        use: 'raw-loader' 
      }
    ]
  }
};

插件plugin

loader 用于转换某些类型的模块，而插件则用于执行范围更广的任务。包括：打包优化，资源管理，注入环境变量

配置文件内多次使用同一插件，需要通过new创建插件实例

const HtmlWebpackPlugin = require('html-webpack-plugin'); // 通过 npm 安装
const webpack = require('webpack'); // 用于访问内置插件
module.exports = {
  plugins: [
    new HtmlWebpackPlugin({template: './src/index.html'})
  ]
};

模式(mode)

  • mode：development, production（默认） 或 none
  • mode可以启用 webpack 内置在相应环境下的优化
    module.exports = {
mode: 'production' 
};

浏览器兼容性(browser compatibility)

  • webpack的import和require.ensure需要Promise
  • 旧浏览器需要提前加载polyfill

    环境(environment)

    webpack 5 运行于 Node.js v10.13.0+ 的版本

    入口起点(entry points)

    单个入口

  • 可以传入文件路径或文件路径数组给entry属性
  • 传入数组则创建multi-main entry，适用于一次注入多个依赖文件，将依赖关系绘制在一个chunk中的情景
  • 单入口可以快速配置webpack，但灵活性不大
    module.exports = {
entry: {
  main: './path/to/my/entry/file.js'
},
// 或
entry: [ 
'./src/file_1.js',
'./src/file_2.js'
],
output: {
  filename: 'bundle.js'
}
};

对象语法

module.exports = {
entry: {
app: ‘./src/app.js’,
adminApp: ‘./src/adminApp.js’
}
};

  • 定义入口的最可扩展的方式
  • 配置可拓展指的是可以重复使用，可以组合配置；将关注点从environment, build target, runtime中分离，并用工具进行合并
  • 通过插件生成入口时，可以传入{}

描述入口的对象

dependOn：当前入口依赖的入口，必须先被加载
filename：输出文件名
import：加载模块
library
runtime：运行时chunk名，未设置将使用入口作为运行时

module.exports = {
  entry: {
    a2: 'dependingfile.js',
    b2: {
      dependOn: 'a2',
      import: './src/app.js'
    }
  }
};

dependOn和runtime不在同一入口同时使用，否则报错
runtime不能指向已存在的入口名，否则报错
dependOn不能循环引用，否则报错

常见场景

分离app和vendor(第三方库)入口

配置两个单独的入口，这样就可以在vendor.js存储未修改的library或文件(例如Bootstrap或jQuery，image)，然后打包成chunk，内容hash不变，浏览器可以独立缓存，减少加载时间

// webpack.config.js
module.exports = {
  entry: {
    main: 'app.js'
    vendor: 'vendor.js'
  }
}
// webpack.prod.js
module.exports = {
  output: {
    filename: '[name].[contenthash].bundle.js'
  }
}
// webpack.dev.js
module.exports = {
  output: {
    filename: '[name].bundle.js'
  }
}
  • w4之前vendor作为单独入口添加到entry，编译成单独文件，w4使用optimization.splitChunks将vendor和app分开，并为vendor创建单独文件，而不为vendor创建entry

多页面应用

  • 多页面应用中，server会拉取新的html文件，页面重新加载并下载资源。
  • 通过这个机会可以使用optimization.splitChunks为页面共享的应用代码创建bundle
  • 多页面应用能复用多个入口之间的代码和模块

输出(output)

output告知weback如何写入编译文件
只能指定一个output配置

用法

// 将一个单独的 bundle.js 文件输出到 dist 目录中

  }

多个入口起点

如果创建出多个chunk，例如多个入口或使用CommonsChunkPlugin插件，需要使用占位符确保文件名称唯一

module.exports = {

    search: './src/search.js'

    filename: '[name].js',
    path: __dirname + '/dist'

// 写入到硬盘：./dist/app.js, ./dist/search.js

loader

loader用于对模块代码进行转换，在import或者load时预处理文件。
因此loader类似于其他构建工具的任务task，并提供了处理前端构建步骤的有效方式
loader将文件从不同语言转为js或将image转为data URL

实现webpack加载css和ts

cnpm install –save-dev css-loader ts-loader

module.exports = {
  module: {
    rules: [
      { test: /\.css$/, use: 'css-loader' },
      { test: /\.ts$/, use: 'ts-loader' }
    ]
  }
};

使用loader的三种方式

  • 配置方式（推荐）：webpack.config.js
    • module.rules指定loader
    • loader从右到左取值并执行
      module.exports = {
module: {
test: '',
use: [
  { loader: 'style-loader'},
  { loader: 'css-loader', options: {modules: true}},
  { loader: 'sass-loader'}
]
}
}
  • 内联方式：import中指定loader
    • import Styles from 'styles-loader!css-loader?modules!./styles.css'
    • ! 将loader分开，每个部分都会相对于当前目录解析
    • !! 将禁用所有已配置的 loader（preLoader, loader, postLoader）
    • 使用 -! 前缀，将禁用所有已配置的 preLoader 和 loader，但是不禁用 postLoaders
    • 选项可以传递查询参数，例如 ?key=value&foo=bar，或JSON 对象，例如 ?{“key”:”value”}

尽可能使用 module.rules，可以减少源码样板文件的代码量，出错时更快地调试和定位 loader 的问题

  • CLI方式：shell命令中指定
    • webpack --module-bind pug-loader --module-bind 'css=style-loader!css-loader'
    • 这会对 .jade 文件使用 pug-loader，对 .css 文件使用 style-loader 和 css-loader

loader特性

  • 支持链式调用
  • 同步或异步都可以
  • 运行在Node
  • 通过options配置（query已废弃）
  • 通过package.json的main导出npm模块作为loader，在module.rules中使用loader引用模块
  • plugin为loader带来更多特性
  • loader能产生额外任意文件

通过loader的预处理函数，更灵活的引入细粒度逻辑，例如压缩打包等

解析loader

loader将从模块路径加载（npm安装，node_nodules加载）
loader模块导出为函数，并且是node兼容的js，按照约定，loader命名为xxx-loader

plugin

plugin目的是解决loader无法实现的事

  • 如果plugin使用了webpack-sources的package，使用require(‘webpack’).sources 替代 require(‘webpack-sources’)

剖析

  • plugin是具有apply方法的js对象，apply会被webpack complier调用，complier在整个生命周期都能访问
  • ConsoleLogOnBuildWebpackPlugin.js
    const pluginName = 'ConsoleLogOnBuildWebpackPlugin';
class ConsoleLogOnBuildWebpackPlugin {
apply(compiler) {
  compiler.hooks.run.tap(pluginName, compilation => {
    console.log('webpack 构建过程开始！');
  });
}
}
module.exports = ConsoleLogOnBuildWebpackPlugin;
  • compiler hook 的 tap 方法的第一个参数，应该是驼峰式命名的插件名称。建议为此使用一个常量，以便它可以在所有 hook 中重复使用。

    模块解析(Modules Resolutions)

    resolver是寻找模块绝对路径的库，模块可以作为另一个模块的依赖被引入
    打包模块时，webpack使用enhanced-resolve解析路径

    解析规则

  • 绝对路径：不做进一步解析
  • 相对路径：相对路径会拼接上下文路径
  • 模块路径：resolve.modules 指定目录检索模块，具体配置参照resolve.alias
    • 如果 package 包含 package.json，那么 resolve.exportsFields 指定的字段会被依次查找
    • package.json 的第一个字段会确定 package 中可用的 export
  • resolver 会检查路径是指向文件还是文件夹
    • 路径指向文件：
      • 如果文件有扩展名，则直接将文件打包。
      • 否则使用 resolve.extensions 选项作为文件扩展名来解析，此选项会告诉解析器在解析中能接受那些扩展名（例如 .js，.jsx）
    • 路径指向文件夹，则寻找具有正确扩展名的文件：
      • 如果文件夹包含 package.json ，则根据 resolve.mainFields的字段顺序查找，并根据 package.json 中的符合配置要求的第一个字段来确定文件路径。
      • 如果不存在 package.json 或 resolve.mainFields 没有返回有效路径，则根据 resolve.mainFiles 指定的文件名顺序查找，看是否能在 import/require 的目录下匹配到一个存在的文件名
      • 然后使用 resolve.extensions ，以类似的方式解析文件扩展名。
      • webpack 会根据构建目标，为这些选项提供合理的默认配置

        解析loader

        loader 的解析规则也遵循特定的规范。但是 resolveLoader 配置项可以为 loader 设置独立的解析规则

        缓存

  • 每次文件系统访问文件都会被缓存，以便于更快触发对同一文件的多个并行或串行请求。
  • 在 watch 模式 下，只有修改过的文件会被从缓存中移出。如果关闭 watch 模式，则会在每次编译前清理缓存

    Module Federation

    动机

    多个独立的构建组成一个应用，独立的构建不存在依赖关系，可以单独开发部署，称为微前端

    底层概念

  • 模块分为本地模块和远程模块，远程模块不属于当前构建，并在运行时从容器中加载
  • 加载远程模块是异步操作，被放置在远程模块和入口之间的下一个chunk的加载操作中
  • chunk的加载操作通过import调用，也支持require.ensure或require()的旧语法
  • 容器由入口创建，入口暴露了对特定模块的异步访问，暴露的访问分为两个步骤：加载异步模块，执行同步模块
    • 加载异步模块在chunk加载期间完成，执行同步模块在其他模块的交错执行期间完成
    • 因此，执行顺序不受模块在本地和远程之间转换的影响
  • 容器可以嵌套使用，之间可以循环依赖

    重载 overriding

  • 容器将本地模块标记为“可重载”，容器的使用者执行重载（提供替换模块）
  • 容器将可重载模块放入一个chunk中，当使用者重写了模块，容器就不再下载对应的可重载模块
  • 替换模块的提供者只提供异步加载函数，容器需要替换模块时才去加载。如果容器不请求则不下载。
  • name用于标识容器中可重载模块，嵌套使用时，相同name的模块将被自动覆盖
  • 重载的提供的步骤类似于容器暴露模块：加载异步和执行同步
  • 必须在容器模块加载前提供重载，初始chunk中只能被不使用Promise的同步模块重载，一旦执行就不可再重载

高级概念

  • 每个构建是一个容器，也可以将其他构建作为容器。所以构建可以通过从容器中加载模块来访问暴露的模块
  • 共享模块指的是可重写、可作为向嵌套容器提供重写的模块。通常指向构建中的相同模块。
  • packageName允许通过设置包名来查找版本，自动推断模块请求。

构建块 (Building blocks)

  • OverridablesPlugin (底层 API)

    • 使特定模块“可重载”。一个本地 API ( __ webpack_override __ ) 允许提供重载。
    • webpack.config.js
      const OverridablesPlugin = require('webpack/lib/container/OverridablesPlugin');
module.exports = {
plugins: [
new OverridablesPlugin([
  {
    // 通过 OverridablesPlugin 定义一个可重载的模块
    test1: './src/test1.js',
  },
]),
],
};
// src/index.js
__webpack_override__({
// 这里我们重写 test1 模块
test1: () => 'I will override test1 module under src',
});

  • ContainerPlugin (底层 API)

    • 该插件使用指定的公开模块来创建一个额外的容器入口。
    • 它还会在内部使用 OverridablesPlugin，并向容器的使用者暴露 override API。

  • ContainerReferencePlugin (底层 API)

    • 该插件将特定的引用添加到作为外部资源（externals）的容器中，并允许从容器中导入远程模块。
    • 它还会调用容器的 override API 来为它们提供重载。
    • 本地的重载（当构建也是容器时，通过 webpack_override 或 override API）和指定的重载被提供给所有引用的容器。

  • ModuleFederationPlugin （高级 API）

    • 该插件组合了 ContainerPlugin 和 ContainerReferencePlugin。
    • 重载（overrides）和可重载（overridables）被合并到指定共享模块的单个列表中

概念目标

  • 既可以暴露，又可以使用 webpack 支持的任何模块类型
  • 代码块加载应该并行加载所需的所有内容 (web到服务器的单次往返)
  • 从使用者到容器的控制
    重写模块是一种单向操作
    同级容器不能重写彼此的模块
  • 概念适用于独立于环境
    可用于 web、Node.js
  • 共享中的相对和绝对请求
    • 会一直提供，即使不使用
    • 会将相对路径解析到 config.context
    • 默认不会使用 requiredVersion
  • 共享中的模块请求
    • 只在使用时提供
    • 会匹配构建中所有使用的相等模块请求
    • 将提供所有匹配模块
    • 将从图中这个位置的 package.json 提取 requiredVersion
    • 当你有嵌套的 node_modules 时，可以提供和使用多个不同的版本
  • 共享中尾部带有 / 的模块请求将匹配所有具有前缀的模块请求

动态远程容器

该容器接口支持 get 和 init 方法。 init 是兼容 async 的方法，只含有一个参数：共享作用域对象(shared scope object)。此对象在远程容器中用作共享作用域，并由 host 提供的模块填充。 可以利用它在运行时动态地将远程容器连接到 host 容器。

init.js

(async () => {
  // 初始化共享作用域（shared scope）用提供的已知此构建和所有远程的模块填充它
  await __webpack_init_sharing__('default');
  const container = window.someContainer; // 或从其他地方获取容器
  // 初始化容器 它可能提供共享模块
  await container.init(__webpack_share_scopes__.default);
  const module = await container.get('./module');
})();

容器尝试提供共享模块，如果共享模块已被使用，则会发出警告，并忽略所提供的共享模块。容器仍能将其作为降级模块。

你可以通过动态加载的方式，提供一个共享模块的不同版本，从而实现 A/B 测试。

Tip：在尝试动态连接远程容器之前，确保已加载容器。

例子：init.js

function loadComponent(scope, module) {
  return async () => {
    // 初始化共享作用域（shared scope）用提供的已知此构建和所有远程的模块填充它
    await __webpack_init_sharing__('default');
    const container = window[scope]; // 或从其他地方获取容器
    // 初始化容器 它可能提供共享模块
    await container.init(__webpack_share_scopes__.default);
    const factory = await window[scope].get(module);
    const Module = factory();
    return Module;
  };
}
loadComponent('abtests', 'test123');

依赖图(dependency graph)

从入口开始，webpack会递归构建一个依赖关系图，包含应用程序所需的模块，然后将所有模块打包为bundle(通常只有一个)，可由浏览器加载

target

target 设置为node，webpack 将在类 Node.js 环境编译代码。(使用 Node.js 的 require 加载 chunk，而不加载任何内置模块，如 fs 或 path)。
每个 target 都包含各种 deployment（部署）/environment（环境）特定的附加项，以满足其需求

module.exports = {
  target: 'node'
};

html插件

webpack-dev-server

  • webpack-dev-server是webpack官方提供的小型Express服务器，为webpack打包生成的资源文件提供web服务。

  • webpack-dev-server 主要提供两个功能：

    • 1.为静态文件提供web服务
    • 2.自动刷新和热替换(HMR)
    • 自动刷新指当修改代码时webpack会自动编译更新内容
    • 热替换指运行时更新各种模块，即局部刷新

  • yarn add webpack-dev-server -D

  • npx webpack-dev-server

  • config.js

    devServer: {
  port: 3000,
  progress: true,
  contentBase: './build' 
}
