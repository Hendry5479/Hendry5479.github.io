# 1. 理念篇

# 1.1 React的设计理念

触发更新：调度

筛选按需更新的组件：协调（diff的官方名称，reconcile）

渲染页面：渲染（DOM,Native）

CPU瓶颈：出现掉帧，解决方法：异步可中断的更新

IO瓶颈：人机交互效果

# 1.2 架构演进

异步可中断的更新：

新增调度阶段，设置任务的优先级，调度和协调过程发生在内存中，不会对视图产生影响。因此可以解决高优中断时的试图展示渲染到一半的页面的情况。

# 1.3 Fiber架构的工作原理

代数效应：用于将副作用从函数中分离

比如异步网络请求的逻辑分离出去，不影响我们关注核心逻辑，关注点变得纯粹。

比如hooks，不关注内部实现逻辑

Fiber用代数效应的思想实现异步可恢复中断的更新，代数效应中try handle示例

概念名词

  • Fiber纤程（可理解为协程的体现，而且可以实现优先级）

  • Process进程

  • Thread纤程

  • Coroutine协程（Generator，传染性 ）

根节点：FiberRootNode

应用根节点：RootFiber

组件节点：FC 节点，Host节点(DOM对应的fiber节点)，文本Fiber节点

子节点用return的指针指向父节点，体现的是递归思想

动态工作单元：

function FiberNode(
  tag: WorkTag,
  pendingProps: mixed,
  key: null | string,
  mode: TypeOfMode,
) {
  // 作为静态数据结构的属性
  this.tag = tag; // 组件类型，比如FC
  this.key = key; //
  this.elementType = null; // 大部分情况与type相同，除了FC被memo包裹这种情况
  this.type = null; // Function， Class，DOM的typeName，
  this.stateNode = null; // 对于HC，真实的DOM

  // 用于连接其他Fiber节点形成Fiber树
  this.return = null;
  this.child = null;
  this.sibling = null;
  this.index = 0;

  this.ref = null;

  // 作为动态的工作单元的属性
  this.pendingProps = pendingProps;
  this.memoizedProps = null;
  this.updateQueue = null;
  this.memoizedState = null;
  this.dependencies = null;

  this.mode = mode;
	// effects
  this.effectTag = NoEffect;
  this.nextEffect = null;

  this.firstEffect = null;
  this.lastEffect = null;

  // 调度优先级相关
  this.lanes = NoLanes;
  this.childLanes = NoLanes;

  // 指向该fiber在另一次更新时对应的fiber
  this.alternate = null;
}

# 递阶段的mount

beginWork(current, workInprogress, renderLanes)

首屏应用渲染， 只有根节点存在current，其他节点只有workInprogress

目的：创建当前fiber节点的第一个子节点，每次调用都只创建一个节点

过程：

  • 判断类型workInProgress.tag ，进入update逻辑的case；

  • updateXXXComponent中会做一些优化，然后调用reconcileChildren，再调用mountChildFibers创建子fiber节点

  • mountChildFibers和reconcileChildFibers的区别：

    ![image-20211029201833927](/Users/hendry5479/Library/Application Support/typora-user-images/image-20211029201833927.png)

    • shouldTrackSideEfftects，如果需要追踪，则给childFiber添加标记；否则直接return
    • 通过current === null判断进入mountChildFibers还是reconcileChildFibers

  • 根据workInProgress fiber是否存在对应的current fiber来决定是否标记effectTag

  • 然后进入reconcile的逻辑，判断当前fiber的children的类型，来创建子fiber节点

render阶段不会执行具体的dom操作，只为需要dom操作的fiber节点打上标记
