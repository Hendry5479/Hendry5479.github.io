    # setState

    # setState的异步和同步更新问题

    只要你进入了 react 的调度流程，那就是异步的。只要你没有进入 react 的调度流程，那就是同步的。

    在生命周期钩子函数中，或者合成事件中，是异步的

    原因：尽可能在接收多个setState后render一次，提升性能

    同步的场景：定时器中或者原生dom事件中

    # setState的数据合并

    多次setState会进行合并，如果想要setState不合并，传入函数即可

    this.setState((state, props) => ({counter: state.counter + 1})

    # shouldComponentUpdate

    render的前提：state、props发生改变

    比如state中有属性m，jsx并没有依赖m，但是m改变了会触发render，这个时候就可以用SCU

    SCU中，改变依赖的属性，返回true；改变m，返回false

    而且，应该所有子组件也使用SCU，否则父组件依赖的state/props改变，子组件仍然会render

    # PureComponent和memo

    SCU的目的是判断props和state的数据是否改变来决定返回true和false

    PureC源码使用了shallowEqual比较state/props，memo使用compare函数比较

    # 数据不可变性

    数据不可以发生改变，否则就是新数据，这样才能触发更新渲染

    如果数据可以改变，但是pureC和memo都不能捕捉到变化，就会产生错误的行为

    # redux

    纯函数：相同输入产生相同输出，没有副作用，例如IO

    redux定义：状态管理容器

    redux要求使用action更新数据（描述更新的type和content的对象）

    强制使用action使状态变化可追踪预测；项目中使用函数定义返回action

    reducer：将state和action联系起来

    reducer定义：将传入的state和action结合，生成新state的纯函数

    三大原则：

    1. 单一数据源：state存储在一个obj tree中，obj tree存储在一个store中
    2. state是只读的：唯一修改state的方式是通过action，不用担心race condition竞态问题
    3. 使用纯函数修改

    实现redux的代码：

    • 创建store保存state：store需要使用redux.createStore(reducer)创建，传入一个reducer
    • 以reducer为主体创建一个store数据库
    • 通过action修改state
    • 完善reducer的代码：reducer是纯函数，不能直接修改state
    • 监听store变化：store.subscribe()

    # 拆分redux

    将store、reducer、action、constants拆分成文件

    # redux流程

    全局只有一个store数据库

    组件派发action

    reducer接收action，返回新state

    state更新后出发通知

    订阅者拿到最新的数据，更新到jsx，发生改变
