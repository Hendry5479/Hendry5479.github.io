                  # 辣鸡回收

                  js引用类型保存在堆内存中，若没有了引用关系，则是不可达对象，需要被清理，否则会内存不足

                  # 回收策略

                  # 标记清除

                  ​ 将活动对象打标记，清除阶段时销毁无标记对象

                  ​ 引擎GC时从根节点（有多个）遍历内存中的对象打标记

                  ​ 分配内存的三种策略：

                  ​ First-fit、Best-fit、Worst-fit

                  ​ 优点：简单

                  ​ 缺点：有内存碎片、慢

                  ​ 标记整理法：标记结束后将活动对象移动到一端，然后清理

                  # 引用计数法

                  ​ 被引用、赋给另一个变量时+1，被覆盖则-1；0则销毁

                  ​ 缺点：1、无法解决循环引用的问题；2、计数器占用内存大

                  function test(){
  let A = new Object()
  let B = new Object()
  
  A.b = B
  B.a = A
}

                  A和B相互引用，则引用数量都是2，不被清理

                  标记清除策略的话，在函数结束后，两个对象都不在作用域中，都被清除掉。

                  # V8的GC优化

                  基于标记清除的分代式GC

                  新生代的特点是存活时间短、比较小

                  因此将内存区域区分为新老生代，使用不同频率的GC，优化性能

                  新生代的GC

                  ​ 将新生代的堆内存分为使用区和空闲区，标记一次后，将活动对象转去空闲区，然后清空使用区，反转

                  ​ 如果多次GC仍然存活，或者对象比较大，则转去老生代

                  老生代GC

                  ​ 不用复制法，直接使用标记清除整理法

                  # 并行回收

                  Stop-The-World：全停顿，JS脚本等待GC后恢复运行的行为

                  并行回收则是在GC时引入辅助线程提高效率

                  依然会停顿，但是会减少耗时，而且不用考虑引用关系的改变

                  新生代GC就采用并行回收

                  # 增量标记和懒清理

                  并行回收依然全停顿，对于老生代而言耗时依然很久

                  # 增量标记：将GC变成可中断

                  中断后恢复的方法：

                  三色标记法：

                  ​ 白色：无活动；灰色：自身标记、成员未标记；黑色：已全标记

                  ​ 每次中断恢复时就从灰色标记开始执行

                  写屏障：

                  ​ 白色对象被引用后还未标记，可能在下一轮被回收的问题

                  ​ 强制让被黑色引用的白色变灰色

                  # 懒清理

                  如果内存足够执行代码，就延迟清理，也不需要全部清理，可以慢慢清

                  增量标记和懒清理减少了STW的时间，但是没有减少主线程的总暂停时间

                  # 并发回收

                  用辅助线程GC，主线程不挂起

                  需要有额外的读写锁机制控制引用关系的变化

                  老生代主要使用并发标记、并发清理，同时用增量的方式分批执行
